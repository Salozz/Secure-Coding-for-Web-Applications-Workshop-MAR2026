🍪 Cookie Security Flags

Every session cookie needs ALL THREE flags. Missing even one creates a vulnerability.

FlagWhat it blocksWithout it...
HttpOnly JavaScript cannot read this cookie (document.cookie hides it). XSS payloads cannot steal the session token. XSS attack on your page → attacker reads document.cookie → gets session token → logs in as victim.
Secure Cookie is ONLY sent over HTTPS. Never transmitted over plain HTTP. On public WiFi, attacker intercepts HTTP request → reads session cookie in plaintext → account taken over.
SameSite=Strict Cookie NOT sent on cross-site requests. CSRF attacks from other websites get no session cookie. CSRF attack (Lab 2A) succeeds — hidden form fires, bank sees valid session, processes transfer. 
// ✅ Complete secure cookie (server-side Node.js):
res.cookie('session', sessionToken, {
  httpOnly:  true,      // No JS access
  secure:    true,      // HTTPS only
  sameSite: 'Strict',   // No cross-site requests
  maxAge:    3600000    // Expires in 1 hour
});

🧠 Quick Check — Answer These Before Revealing

1. Why is bcrypt better than MD5 for password hashing? Give one specific technical reason.

bcrypt is intentionally SLOW — designed to take ~100-200ms per hash. This makes brute-force attacks impractical. MD5 was designed for SPEED (file checksums), which means attackers can try billions of guesses per second on modern GPUs.

2. If a cookie has HttpOnly=true, can an XSS attack steal the session token?

No. HttpOnly means document.cookie will NOT return that cookie. JavaScript — including injected XSS payloads — cannot read it. This is why defence-in-depth matters: even if XSS gets through your sanitisation, HttpOnly limits the damage.

3. What attack does SameSite=Strict prevent, and how?

SameSite=Strict prevents CSRF. When set, the browser ONLY sends the cookie on requests that originate from the same site. A hidden form on evil.com that posts to bank.com gets NO session cookie — the bank server sees an unauthenticated request and rejects it.
