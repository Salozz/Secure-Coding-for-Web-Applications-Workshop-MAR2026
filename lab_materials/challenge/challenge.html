🖥 The Application

📄 Source Code to Review

<!-- The login form -->
<form method="GET" action="/login">  <!-- ISSUE? -->
  <input type="text"     id="user" placeholder="Username">
  <input type="password" id="pass" placeholder="Password">
  <button>Login</button>
</form>

<script>
  // Display welcome message from URL
  const name = new URLSearchParams(window.location.search).get('name');
  document.getElementById('welcome').innerHTML = 'Hello ' + name;  // ISSUE?

  // Create session token
  const SESSION = 'user_' + Date.now();  // ISSUE?

  // Set session cookie
  document.cookie = 'session=' + SESSION;  // ISSUE?
  // Missing: HttpOnly, Secure, SameSite

  // Password check
  const PASSWORDS = { admin: 'admin123', user: 'password' };  // ISSUE?

  function checkLogin(e) {
    e.preventDefault();
    const u = document.getElementById('user').value;
    const p = document.getElementById('pass').value;
    if (PASSWORDS[u] === p) {  // ISSUE?
      window.location.href = '/dashboard?user=' + u;
    }
  }
</script>

📋 Answer Key

Issue 1: GET method for credentials

Password appears in the URL, server logs, browser history, and Referer headers. Fix: use method="POST". POST params go in the request body, not the URL.

Issue 2: XSS via innerHTML with URL parameter

The ?name= URL parameter is inserted as HTML. Attacker crafts a malicious link → victim's browser executes it. Fix: use textContent instead of innerHTML.

Issue 3: Predictable session token

Date.now() is a millisecond timestamp — an attacker who knows when you logged in can guess the token. Fix: use crypto.randomBytes(32).toString('hex') on the server.

Issue 4: Cookie with no security flags

No HttpOnly (JS can read it → XSS steals session), no Secure (sent over HTTP), no SameSite (CSRF works). Fix: set all three flags server-side. Never create session cookies from client JS.

Issue 5: Credentials in client-side code

The PASSWORDS object is in the JavaScript source — visible to anyone via DevTools → Sources or View Page Source. Fix: authentication must ALWAYS happen server-side. The client never sees stored passwords.

Issue 6: Client-side authentication logic

The login check is in the browser. Any user can open the console and type window.location.href='/dashboard?user=admin' to bypass it completely. Fix: ALL auth decisions are server-side. Server validates session token on every protected request.

Bonus Issue 7: Open redirect

window.location.href='/dashboard?user=' + u — if the attacker controls the value of u (e.g. via URL manipulation), they could redirect to a malicious site. Validate and sanitise all redirect targets.
